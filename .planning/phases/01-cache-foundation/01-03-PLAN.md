The task has two passes:

**Pass 1 -- Repo Trends (deduplicated by repo_id):**
- Use `processRepos` from the task context (it iterates distinct repos, not per-project). Study `build-static-api.task.ts` to see how it uses `processRepos` -- it calls a callback for each repo with the repo data including snapshots and related projects.
- Filter: only process repos linked to at least one project with `status IN ('active', 'featured', 'promoted')`. Use `inArray` from drizzle-orm. If `processRepos` doesn't support a where clause, query distinct repo IDs first and filter within the callback.
- For each eligible repo:
  1. Get snapshots via `flattenSnapshots(repo.snapshots)` -- snapshots come from the repo data
  2. Compute trends via `computeTrends(flattenedSnapshots)` -- reuse existing function
  3. Compute `popularityScore` via `computePopularityScore(trends)`
  4. Compute `activityScore` via `computeActivityScore(repo.last_commit, repo.contributor_count)`
  5. Upsert into `repoTrends`: `db.insert(schema.repoTrends).values({...}).onConflictDoUpdate({ target: schema.repoTrends.repoId, set: {...} })`
  6. Map `undefined` from `computeTrends()` to `null` for SQL (use `trends.daily ?? null`)
- Use individual upserts per repo (not batch) per RESEARCH.md recommendation -- simpler and fast enough for ~3K rows

**Pass 2 -- Project Trends (per project):**
- Use `processProjects` from the task context or query projects directly with status filter: `WHERE status IN ('active', 'featured', 'promoted')`
- For each eligible project:
  1. Get the project's packages (from the project data or a separate query)
  2. Call `resolvePrimaryPackage(packages)` to get the primary package (highest downloads)
  3. Look up the repo's scores from the repo_trends row just written in Pass 1 (either from a Map built during Pass 1, or query `repoTrends` by `project.repoId`)
  4. Compute `usageScore` via `computeUsageScore(primaryPackage?.monthlyDownloads ?? null)`
  5. Compute `relevanceScore` via `computeRelevanceScore(repoPopularityScore, repoActivityScore, usageScore, !!primaryPackage)`
  6. Upsert into `projectTrends`: `db.insert(schema.projectTrends).values({...}).onConflictDoUpdate({ target: schema.projectTrends.projectId, set: {...} })`

**Performance optimization:** During Pass 1, build a `Map<repoId, { popularityScore, activityScore }>` so Pass 2 can look up repo scores without querying the database again.

**Return value:**
```typescript
return { data: null, meta: { repos: repoCount, projects: projectCount } };
```

**Error handling:** Use try/catch per item within the process callback. Log errors via `logger.error()` and continue processing remaining items. Track error count in meta.

**Logging:** Log summary stats: `logger.info("Refreshed {N} repos and {M} projects")`. Optionally log score distribution (min/max/avg) for tuning visibility -- this is Claude's discretion per CONTEXT.md.

IMPORTANT: Do NOT use `getPackageData()` from project-helpers.ts for primary package resolution -- it picks `packages[0]` (first) not highest downloads. Use `resolvePrimaryPackage()` from Plan 02.

IMPORTANT: Map `computeTrends()` undefined outputs to `null` for SQL. `undefined` in Drizzle may be interpreted as "don't set this column" rather than SQL NULL.
